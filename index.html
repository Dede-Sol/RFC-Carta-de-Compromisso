Resumo Executivo
Este documento propõe a automação completa do processo de emissão de Cartas de Compromisso na Ouvidoria do Nubank. Hoje, o analista cola manualmente dados do Marte em uma planilha Excel, que agrupa transações por adquirente ou TED/Boleto e gera cartas no Google Drive. Com IA, esse fluxo será substituído por um pipeline que lê os dados automaticamente, classifica as transações, preenche os templates e envia os documentos — reduzindo o tempo de ciclo estimado em 70% e eliminando erros de preenchimento.

1. Fluxo To-Be — Sistema com IA

O novo fluxo elimina todas as etapas manuais de copiar e colar. A IA assume a leitura, classificação e preenchimento; o analista só revisa e aprova.

1.1 Visão geral do fluxo

#EtapaDescrição
1Gatilho automáticoUm caso de reclamação é aberto no Marte ou em fila monitorada. O sistema de IA é acionado via webhook ou polling periódico.
2Extração de dadosA IA lê os campos do caso: transações, valores, datas, CPF/CNPJ e Customer ID. Nenhum "copiar e colar" é necessário.
3Classificação automáticaO modelo identifica o tipo de transação: (a) Adquirente ou (b) TED/Boleto. Isso define qual template e qual fluxo de agrupamento será usado.
4Lookup de IFO sistema consulta a base "Contato de Bancos" (158+ IFs mapeadas) para localizar ISPB, e-mail principal e e-mail cópia da IF destinatária.
5Agrupamento de transaçõesTransações do mesmo adquirente (ou mesmo banco recebedor) são consolidadas em um único grupo — equivalente às abas 1.1/2.1 da planilha atual.
6Geração da cartaO template é preenchido automaticamente com: nome do cliente, CPF/CNPJ, Customer ID, lista de transações, valores e datas. O documento é criado no Google Drive.
7Revisão humanaO analista recebe uma notificação com link para a carta gerada. Ele pode aprovar com 1 clique, editar campos ou rejeitar (com comentário).
8Envio da cartaApós aprovação, o sistema envia automaticamente via legitimidade@nubank.com.br para o(s) e-mail(s) da IF. O ticket é atualizado com comprovante.
9Registro e auditoriaTodos os eventos (geração, aprovação, envio) são gravados com timestamp, ID do analista e versão do documento para fins regulatórios.
10IFs não mapeadasSe o adquirente não constar na base, o sistema sinaliza, notifica o analista e abre uma tarefa para incluir o novo contato na base.

2. Plano de Implantação em Fases

A implantação segue a lógica "caminhar antes de correr": cada fase entrega valor real e serve de aprendizado para a próxima.

FASE 1 — MVP Semanas 1–6 Automação da extração e preenchimento básico
  • Criar script Python que lê o CSV exportado do Marte (sem integração live ainda).
  • Script classifica transações em Adquirente ou TED/Boleto usando regras simples (if/else).
  • Lookup automático na aba "Contato de Bancos" pelo ISPB.
  • Gerar carta em Google Docs via API (usando o ID do template existente).
  • Salvar carta na pasta do Drive com nome padronizado.
  • Enviar e-mail com anexo via Gmail API para o endereço da IF.
  • Log de ações em planilha de auditoria simples.
✅ Meta: Analistas testam o script em casos reais de baixo risco. Feedback semanal.
FASE 2 — Integração Marte Semanas 7–12 Eliminar a etapa de exportação manual
  • Integração com a API do Marte para leitura direta dos dados do caso.
  • Gatilho automático quando caso entra em status "Carta de Compromisso".
  • Dashboard simples mostrando cartas pendentes, aprovadas e enviadas.
  • Implementar tela de revisão com diff visual (o que a IA preencheu vs. template).
  • Tratamento automático de IFs não mapeadas com alerta ao analista.
✅ Meta: Piloto com 2–3 analistas. 80% das cartas sem intervenção manual.
FASE 3 — IA Generativa Semanas 13–20 Usar LLM para adaptar linguagem e detectar inconsistências
  • Integrar modelo de linguagem (ex.: Claude API) para redigir parágrafos personalizados.
  • Detecção automática de inconsistências nos dados (valor negativo, data futura, etc.).
  • Sugestão de e-mail de encaminhamento ao cliente após envio da carta.
  • Classificação de risco do caso (baixo/médio/alto) para priorização.
  • Modelo aprende com correções dos analistas (feedback loop).
✅ Meta: Expansão para toda a equipe de Ouvidoria. Monitoramento de acurácia semanal.
FASE 4 — Expansão Semanas 21–30 Integrar com outros sistemas e otimizar a operação
  • Integração com sistema de protocolo do BACEN/PROCON para prazos automáticos.
  • Alertas de SLA: carta não enviada a X horas do prazo regulatório.
  • Relatórios automáticos para gestores (volume, tempo médio, erros).
  • Atualização automática da base de IFs via consulta ao ISPB do BACEN.
  • Avaliação de expansão para outras áreas (ex.: Contestações, Fraude).
✅ Meta: Operação plena. Revisão trimestral de performance e ajustes de modelo.

3. Riscos Regulatórios e Controles

A Carta de Compromisso é um documento com validade jurídica. A automação deve respeitar obrigações do BACEN e do CDC.

ID Risco Impacto Prob. Descrição Controle
R1 Documento com conteúdo incorreto Alto Alto IA preenche campo errado (ex.: CPF trocado entre clientes). Revisão humana obrigatória antes do envio. Double-check automático de CPF/CNPJ. Diff visual.
R2 Envio para IF errada Alto Médio Lookup retorna e-mail incorreto por ISPB desatualizado. Validação cruzada do ISPB com a API do BACEN. Alerta se ISPB não encontrado.
R3 Perda de prazo regulatório Alto Médio Falha no sistema gera atraso no envio dentro do prazo legal. SLA automatizado com alerta 24h antes do prazo. Fallback manual com notificação ao gestor.
R4 Violação da LGPD Alto Baixo Dados de clientes processados por IA sem base legal clara. Revisar com DPO se o uso de IA está coberto pela política de privacidade. Logs de acesso.
R5 Adulteração do template Médio Baixo IA modifica cláusulas jurídicas do template sem validação. Template versionado e bloqueado. IA só preenche campos variáveis. Aprovação pelo Jurídico.
R6 Acesso não autorizado Médio Médio Terceiros acessam cartas ou dados de clientes via falha de permissão. Controle de acesso baseado em papéis (RBAC). Pasta do Drive com permissão restrita.
R7 Alucinação do modelo de IA Alto Médio LLM inventa dados ou cláusulas que não existem no template original. IA usa APENAS campos pré-definidos. Nenhum texto livre gerado sem revisão. Testes mensais.
R8 Continuidade operacional Médio Baixo Sistema fora do ar impede geração de cartas dentro do prazo. Fallback documentado: planilha atual continua disponível como contingência. Runbook de incidente.

3.1 Frameworks Regulatórios Aplicáveis

FrameworkRelevância para o projeto
BACEN — Resolução CMN 4.860/2020Regula o funcionamento dos componentes do sistema de pagamentos. As cartas de compromisso são documentos formais dentro desse ecossistema.
LGPD (Lei 13.709/2018)Dados pessoais de clientes (CPF, nome, Customer ID) são processados no fluxo. É preciso confirmar com o DPO se o uso de IA está previsto nas políticas internas.
Código de Defesa do ConsumidorPrazos de resposta a reclamações são regulados. A automação deve garantir cumprimento, não criar novos riscos de atraso.
Política interna — Segurança da InformaçãoVerificar com o time de InfoSec se o uso de APIs externas (ex.: LLMs) é permitido para dados de clientes ou se deve ser via modelo interno.

4. Playbook Operacional

Este playbook é o guia do dia a dia para os analistas que irão operar o sistema. Mantenha-o atualizado a cada iteração do produto.

4.1 Como usar o sistema — Fase 1 (MVP)

  1. Abra o caso no Marte e exporte o CSV de transações (botão "Exportar dados").
  2. Acesse o painel da ferramenta em [URL a definir] e faça o upload do CSV.
  3. O sistema classifica automaticamente: confirme se é "Adquirente" ou "TED/Boleto".
  4. Revise os campos preenchidos: nome do cliente, CPF, valores e e-mail da IF.
  5. Se tudo estiver correto, clique em "Aprovar e Enviar". Se não, clique em "Editar".
  6. Verifique no Google Drive que a carta foi salva na pasta correta.
  7. Confirme o recebimento do e-mail de confirmação de envio.
  8. Em caso de IF não mapeada: preencha o formulário de cadastro de nova IF.

4.2 Regras de negócio importantes

  • O e-mail de envio deve ser sempre legitimidade@nubank.com.br — nunca o e-mail pessoal do analista.
  • Cartas para o mesmo adquirente (mesmo ISPB) devem ser consolidadas em um único documento.
  • O Customer ID deve ser conferido manualmente se houver mais de uma pessoa no caso.
  • Documentos enviados não podem ser alterados — abra novo caso para correções.
  • IFs não mapeadas devem ser cadastradas em até 24h para não bloquear o fluxo.

4.3 O que fazer se algo der errado (Troubleshooting)

ProblemaO que fazer
Sistema não reconhece a IFVerifique se o ISPB está correto. Se a IF não está na base, cadastre-a pela tela de "Nova IF".
Carta gerada com dados erradosClique em "Rejeitar", corrija os dados no formulário e gere novamente. Não edite o arquivo no Drive diretamente.
E-mail de envio falhouAcesse o log de erros no painel. Tente reenviar. Se persistir, envie manualmente e registre o incidente.
Sistema fora do arUse a planilha Excel de contingência (Carta de Compromisso v2). Notifique o time de suporte via Slack.
Prazo regulatório próximoPriorize os casos com alerta vermelho no painel. Se necessário, envie manualmente e registre na planilha de auditoria.

4.4 Glossário rápido para iniciantes

TermoSignificado
ISPBCódigo de identificação de uma Instituição Financeira no sistema do BACEN (ex.: 0 = Banco do Brasil).
IFInstituição Financeira — o banco ou adquirente que receberá a carta.
AdquirenteEmpresa que processa pagamentos com cartão (ex.: Cielo, Stone, Getnet).
TED/BoletoTransações via Transferência Eletrônica Disponível ou boleto bancário.
MarteSistema interno do Nubank onde ficam registradas as reclamações dos clientes.
Carta de CompromissoDocumento formal enviado à IF solicitando posicionamento sobre uma transação contestada.
LGPDLei Geral de Proteção de Dados — regula como dados pessoais podem ser usados.
LLMLarge Language Model — o tipo de IA (como o Claude) que lê e gera texto com contexto.
MVPMinimum Viable Product — a versão mais simples do sistema que já entrega valor real.
WebhookMecanismo automático que avisa um sistema quando um evento acontece em outro sistema.

4.5 Dicas para quem está aprendendo Python

Se você quiser entender ou adaptar o código do sistema, aqui estão os pontos de partida:

pandas — lê e manipula tabelas (como Excel). Você já usa o conceito — é a mesma lógica das abas da planilha.
import pandas as pd   →   df = pd.read_excel("arquivo.xlsx")
requests / httpx — para chamar APIs externas (ex.: Google Drive, Marte). É como "pedir uma informação para outro sistema".
import requests   →   resposta = requests.get("https://api.exemplo.com/dados")
python-docx — cria e edita documentos Word por código. Útil para gerar templates automaticamente.
from docx import Document   →   doc = Document("template.docx")
google-api-python-client — interage com Google Drive e Gmail diretamente pelo Python.
from googleapiclient.discovery import build   →   service = build("drive", "v3", credentials=creds)
💡 Comece pelo script da Fase 1: ele não precisa de nenhuma integração — só lê um CSV e gera a carta. É o melhor ponto de entrada para quem está aprendendo Python.
